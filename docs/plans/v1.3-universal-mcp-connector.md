# StoryDeck v1.3 — Universal MCP Connector (Product Plan)

**Status:** Proposed  
**Date:** 2026-07-22  
**Audience:** Product + engineering  
**Builds on:** v1.2 MCP server (`src/mcp/`), `runtime.json`, `install-mcp.sh`

---

## The user problem (reframed correctly)

A user opens StoryDeck. They use **some** AI harness — Cursor, Claude Desktop, Claude Code, OpenCode, Zed, Windsurf, Kimi Code, VS Code + Cline, etc. They want their agent to **see and edit their live board**.

Today they must:
- Know what MCP is
- Find a shell script or README
- Edit JSON in hidden config folders
- Restart the harness manually

**That is not a product.** v1.2 shipped the plumbing; v1.3 ships the **front door**.

---

## What MCP actually is (one sentence for the UI)

> **StoryDeck is the MCP server.** Your AI app is the client.  
> “Connect” means: *register StoryDeck inside your AI app so it can talk to this board.*

This is **not** the same as the in-app **AI key** button (that powers StoryDeck’s own ask bar). MCP connects **external** agents.

---

## Product vision: “Connect through MCP”

### Entry point (in StoryDeck header)

Add a button next to **AI key**:

**`Connect AI`** (or **`MCP`**) → opens **Universal Connector** modal.

### One-click “Connect” does (autonomously)

1. Detect harness install + config file path for this OS
2. Merge `storydeck` entry into that client’s MCP config (same logic as `install-mcp.sh`, generalized)
3. Write `STORYDECK_RUNTIME_FILE` pointing at userData
4. Show success: **“Connected — restart Cursor to activate”**
5. Optionally offer **“Open Cursor”** (best-effort `open -a Cursor`)

**User never runs a terminal script.**

## Architecture

```
┌──────────────────────────────────────────────────────────────┐
│  StoryDeck app (Electron)                                    │
│  ┌─────────────┐   IPC/API    ┌──────────────────────────┐   │
│  │ Connect AI  │ ──────────► │ mcp-connector (main proc) │   │
│  │ modal (web) │             │ · detect clients          │   │
│  └─────────────┘             │ · merge configs           │   │
│                              │ · verify connection       │   │
│                              └────────────┬─────────────┘   │
│  ┌─────────────┐                          │ writes           │
│  │ MCP server  │◄── stdio ────────────────┼── harness proc   │
│  │ (existing)  │         (spawned by      │                  │
│  └──────┬──────┘          client app)     ▼                  │
│         │ HTTP                    ~/.cursor/mcp.json         │
│         ▼                         ~/.../claude_desktop_...   │
│     todo.db                       ~/.config/opencode/...     │
└──────────────────────────────────────────────────────────────┘
```

### New modules (proposed)

| Module | Role |
|--------|------|
| `src/mcp/registry.js` | Catalog of supported harnesses + paths per OS |
| `src/mcp/connector.js` | Detect, connect, disconnect, status |
| `src/mcp/detect.js` | Is app installed? Is storydeck entry present? |
| `main.js` hooks | IPC or `/api/mcp/*` endpoints (config writes need main process) |

**Why main process:** packaged app knows its own `MCP_SCRIPT` path and `userData`; renderer must not write arbitrary files.

## Harness registry (v1.3 launch set)

Each entry:

```js
{
  id: 'cursor',
  name: 'Cursor',
  detect: { darwin: ['/Applications/Cursor.app'], win32: [...], linux: [...] },
  config: { darwin: '~/.cursor/mcp.json', mergeKey: 'mcpServers.storydeck' },
  command: { packaged: 'electron-as-node', dev: 'node' },
  restartHint: 'Restart Cursor',
  docsUrl: 'https://docs.cursor.com/context/mcp',
}
```

### Phase 1 clients (macOS first — matches current user base)

| Client | Config location (macOS) | Detect app |
|--------|-------------------------|------------|
| **Cursor** | `~/.cursor/mcp.json` | `/Applications/Cursor.app` |
| **Claude Desktop** | `~/Library/Application Support/Claude/claude_desktop_config.json` | `/Applications/Claude.app` |
| **Windsurf** | `~/.codeium/windsurf/mcp_config.json` (verify) | Windsurf.app |
| **OpenCode** | `~/.config/opencode/opencode.json` or project config (verify) | CLI + config presence |
| **Claude Code** | `~/.claude/settings.json` or MCP section (verify) | `claude` CLI |
| **Zed** | `~/Library/Application Support/Zed/settings.json` | Zed.app |
| **Manual / Other** | N/A | Show copy-paste JSON + instructions |

Phase 2: Windows + Linux paths for same clients.  
Phase 3: VS Code (`mcp.json` in user profile), Cline, Kimi Code (as docs/paths stabilize).

**Important:** registry paths must be verified per client before ship; wrong path = broken one-click.

## API (in-app)

| Endpoint | Purpose |
|----------|---------|
| `GET /api/mcp/harnesses` | List all + `{ installed, connected, configPath }` per harness |
| `POST /api/mcp/connect` | `{ harnessId }` → merge config, return `{ ok, restartHint }` |
| `POST /api/mcp/disconnect` | Remove `storydeck` entry (optional v1.3.1) |
| `GET /api/mcp/manual?harness=other` | Generated JSON snippet for unsupported clients |
| `POST /api/mcp/test` | Probe MCP server + live API (same as `storydeck_status`) |

UI polls `GET /api/mcp/harnesses` when modal opens.

---

## UX rules (non-negotiable)

1. **Never ask the user to run a shell script** for MCP setup.
2. **Never send them to README** for the primary path — in-app only.
3. **Always show connection status** (connected / needs restart / not installed).
4. **Copy-paste fallback** for unknown harnesses — still one modal, not external docs.
5. **Autonomous deploy** — connector ships via hot-update where possible; registry tweaks don’t need reinstall.
6. **Distinguish AI key vs MCP** in the modal copy (two different features).

## Connection states (per harness)

| State | UI | Meaning |
|-------|-----|---------|
| `not_installed` | Grey card | App not found on disk |
| `available` | **[Connect]** button | App found, not configured |
| `connected` | Green ✓ **Connected** | Config contains valid `storydeck` entry |
| `needs_restart` | Amber ⚠ | Config written this session; client not reloaded |
| `error` | Red | Config write failed (permissions, parse error) |

## “Test connection” (in modal)

Runs locally (no cloud):

1. Confirm `runtime.json` exists
2. Hit `/api/version` on loopback
3. Optionally spawn MCP server once and call `storydeck_list` (advanced)

Shows: **“Ready — 73 pending stories”** or **“Start StoryDeck first”**.

---

## What this is NOT

| Misconception | Reality |
|---------------|---------|
| “StoryDeck hosts MCP clients inside the app” | Clients run in Cursor/Zed/etc.; StoryDeck only registers itself |
| “One protocol connects all AIs automatically” | Each harness has its own config file; we maintain a registry |
| “User picks MCP server from a global directory” | There is no OS-wide MCP store; we **push** StoryDeck into each client |
| “Replaces AI key” | AI key = built-in ask bar; MCP = external agents |

## Phased delivery

### v1.3.0 — Universal Connector (MVP)

- **Connect AI** header button + modal
- Registry: Cursor + Claude Desktop (macOS)
- `GET/POST /api/mcp/*` in main process
- Migrate logic from `install-mcp.sh` → `connector.js`
- Tests: config merge, detect, no secret leakage
- Hot-updateable: `web/` modal + `src/mcp/registry.js`

**Binary need:** IPC/file-write from API (small `main.js` addition) — one reinstall, then hot-fixes.

### v1.3.1 — Expand harnesses

- OpenCode, Zed, Windsurf, Claude Code
- Windows paths
- Disconnect + reconnect

### v1.4 — Polish

- “Open app” buttons after connect
- First-run tooltip: “Connect your AI?”
- Optional: detect when harness restarted (watch config mtime — best effort)

## Risks

| Risk | Mitigation |
|------|------------|
| Client config format changes | Versioned registry; hot-update registry without app reinstall |
| Wrong config path breaks user’s client JSON | Merge-only (never replace whole file); backup before write |
| Sandboxed Mac blocks writes to `~/.cursor` | Run writes from Electron main (same as today’s installer) |
| User expects in-app chat = MCP | Clear copy in modal + tooltips |

---

## Open questions

1. **Button label:** “Connect AI” vs “MCP” vs “Agents” — user test (recommend **Connect AI** with subtitle “via MCP”).
2. **Kimi / niche clients:** registry plugin file users can drop in `userData/mcp-harnesses.json`?
3. **Deep links:** if clients add `cursor://mcp/install?...` later, add second one-click path.

---

## Recommendation

**Yes — build the Universal Connector in v1.3.** v1.2 was engineer-facing; v1.3 is user-facing. Same MCP server, new **discovery + one-click install UI** in StoryDeck, modeled after the existing **AI key** modal pattern users already understand.

**Next step when approved:** implement Phase 1 (Cursor + Claude Desktop, macOS) without asking the user to run any scripts.
