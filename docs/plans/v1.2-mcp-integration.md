**Status:** Implemented (v1.2.0 — pending binary install on device)  
**Author:** /team plan  
**Date:** 2026-07-22  
**Target release:** v1.2.0 (binary) + content hot-update channel  
**Audience:** Maintainer (Rijul), Cursor MCP users

---

## Executive Summary

StoryDeck v1.1 runs a local HTTP API on loopback but has **no MCP bridge**, so Cursor cannot natively read or mutate the live board. **v1.2 adds a stdio MCP server** that proxies to the running app, plus **port discovery** so agents find the correct localhost port after fallback binding.

**Update model (user-facing):**

| Change type | Auto-updates? | User action |
|-------------|---------------|-------------|
| UI + server JS (`web/`, `src/`) | Yes — on next relaunch | Quit & reopen StoryDeck |
| Electron binary (`main.js`, deps) | No | Reinstall `.dmg` or run `./tools/update-app.sh` |
| Cursor MCP config | One-time | Run `./tools/install-mcp.sh` once |

v1.2 is primarily a **binary release** (port file in `main.js`). MCP server code lives in hot-updatable `src/mcp/` so post-1.2 tool tweaks can ship without reinstall.

---

## Goals

1. **G1 — Cursor-native board access:** MCP tools list, create, update, complete, comment on, and export stories from the **installed** app while it is running.
2. **G2 — Zero cloud:** MCP talks only to `127.0.0.1`; no new outbound data paths.
3. **G3 — Port resilience:** MCP works when StoryDeck binds a fallback port (EADDRINUSE → OS-assigned port).
4. **G4 — One-time setup:** Installer script writes Cursor MCP config; documented manual fallback.
5. **G5 — Clear update story:** README + release notes explain hot-update vs reinstall.

## Non-Goals (v1.2)

- **NG1:** Binary auto-update via `electron-updater` (defer to v1.3+).
- **NG2:** MCP over the network / remote boards.
- **NG3:** Replacing the in-app AI assistant with MCP.
- **NG4:** Windows/Linux MCP installer polish beyond a documented JSON snippet (macOS-first).

### Port discovery (new in v1.2 binary)

**Problem:** `main.js` may bind port `0` when `4321` is taken; MCP cannot hardcode `4321`.

**Solution:** On successful listen, write `userData/runtime.json`:

```json
{
  "host": "127.0.0.1",
  "port": 4321,
  "pid": 52777,
  "startedAt": "2026-07-22T18:20:00.000Z",
  "appVersion": "1.2.0"
}
```

- Written in `main.js` after `listenWithFallback` succeeds.
- Removed on `app.quit` / `window-all-closed`.
- MCP server reads this file first; falls back to `STORYDECK_URL` env, then `http://127.0.0.1:4321`.
- Stale file detection: if `GET /api/version` fails, report *"StoryDeck is not running"* with setup hint.

### MCP server placement

| File | Hot-updatable? | Notes |
|------|----------------|-------|
| `src/mcp/server.js` | Yes | stdio entrypoint |
| `src/mcp/client.js` | Yes | HTTP wrapper + error types |
| `src/mcp/tools.js` | Yes | Tool definitions + handlers |
| `main.js` (runtime.json) | **No** | Requires v1.2 binary |
| `tools/install-mcp.sh` | Repo only | Writes `~/.cursor/mcp.json` merge |

**Dependency:** Add `@modelcontextprotocol/sdk` to `package.json` dependencies (ships in app bundle).

## MCP Tool Specification

| Tool | Description | API mapping |
|------|-------------|-------------|
| `storydeck_status` | App version, content version, story counts, connection URL | `GET /api/version` + `GET /api/state` (counts only) |
| `storydeck_list` | List stories; filters: `status`, `project` (epic), `search` | `GET /api/state` + client-side filter |
| `storydeck_get` | Get one story by id | `GET /api/stories/:id` |
| `storydeck_create` | Create story (`task` required; optional `project`, `points`, `due`, `note`) | `POST /api/stories` |
| `storydeck_update` | Patch fields on a story | `PATCH /api/stories/:id` |
| `storydeck_complete` | Mark story done (convenience wrapper) | `PATCH` with `status: done` |
| `storydeck_comment` | Add comment to a story | `POST /api/comments` |
| `storydeck_delete` | Delete a story | `DELETE /api/stories/:id` |
| `storydeck_export` | Export full board JSON | `GET /api/export` |

**Intentionally excluded from v1.2:** `storydeck_import`, `storydeck_reset` (destructive bulk ops — add in v1.2.1 if needed with confirmation flag).

### Error contract

| Condition | MCP response |
|-----------|--------------|
| App not running | Clear error: *"Start StoryDeck first"* + path to app |
| Story not found | 404 mapped to tool error with story id |
| Validation error (empty task) | Pass through API `error` message |
| Non-loopback URL configured | Refuse connection (security) |

## Security Considerations

1. **Loopback only** — MCP client rejects non-`127.0.0.1` / `::1` hosts (mirrors `server.js` guard).
2. **No auth token in v1.2** — acceptable because API is already localhost-only; document threat model.
3. **No import/reset tools** — reduces risk of accidental board wipe via agent.
4. **Runtime file** — contains port/pid only; no secrets.

## Risks & Mitigations

| Risk | Impact | Mitigation |
|------|--------|------------|
| Cursor MCP config path differs by install | Installer fails silently | Document manual JSON; detect `~/.cursor/mcp.json` vs project-level |
| Port file stale after crash | MCP connects to wrong port | Probe `/api/version`; TTL check on `startedAt` + pid alive |
| `@modelcontextprotocol/sdk` in asar | `node` can't load from asar | Add `src/mcp/**` to `asarUnpack` in electron-builder config |
| v1.1 users expect MCP without reinstall | Confusion | runtime.json requires binary; README states minimum v1.2 |

## Appendix B — Update FAQ (for release notes)

**Q: Do I need to reinstall for every update?**  
A: No. UI and server fixes hot-update when you relaunch. Only app-shell changes (like v1.1 → v1.2) need a new `.dmg`.

**Q: Will MCP config auto-install?**  
A: No. Run `./tools/install-mcp.sh` once after upgrading to v1.2.
