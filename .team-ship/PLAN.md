# Ship Plan — todo.html → local SQLite app + search + compact view + Anthropic UI audit

Generated by `/team:ship`. Adapted (workspace is not a git repo): no PR/CI stages; hard
verification gates retained (real tests, independent verify, claim↔evidence).

## Stage 1 — Research / audit (DONE)

Two read-only agents audited the codebase and authored a design spec.

**Code/UI audit (key facts):**
- Single file `todo.html`: CSS 11–472, HTML 474–515, JS 517–1418. All state + handlers global; inline `onclick`/`onchange`.
- Persistence choke point: `save()` (623–625) writes the whole `tasks` array; called by ~9 mutators.
- SEED (519–591, 71 stories) is a **runtime dependency**, not just seed data — used as the 3-way-merge baseline and auto-reinserted in `runMigrations`. Must be untangled for a DB.
- Ordering is **implicit array index**, global (not per-column), rebuilt from DOM on drop. No `position` field.
- IDs are client-generated `max(id)+1` — unsafe for a DB (DB must own ids).
- Story shape: `{id, task, project(epic, ''=Unassigned), status(pending|done), added, completed?, workStatus?(in-progress|blocked), points, urgent?, note?, comments[{text,created}], sub?(dead)}`.
- Sprint state is derived: done > blocked > in-progress > todo (`getSprintState`).

**Dropdown audit:** native `<select>` (modal `#new-task-project`; card `.epic-select`) lack `appearance:none`, custom chevron, focus ring, hover — OS chrome clashes with theme. Native checkbox (`#new-task-urgent`) is OS-blue. Number spinner + scrollbars + missing `:focus-visible` rings on buttons are also off-theme. Placeholder text uses `--faint-text` (#8e8b82) which FAILS WCAG AA — must use `--muted`.

## Stage 2/3 — Architecture decision + plan

### Decision: local-first, on-device only
- **Runtime:** Node's built-in `node:sqlite` (`DatabaseSync`) — verified working on Node v25.8.2 with NO flags and NO native compile. Avoids `better-sqlite3` build risk entirely.
- **Server:** minimal HTTP server (Node stdlib `http`, zero deps) bound to **`127.0.0.1` only** — refuses non-loopback. Serves the frontend + JSON REST API.
- **DB file:** `data/todo.db` on local disk. Backups: JSON export/import retained + timed snapshot to `backups/`.
- **Frontend:** reuse existing Anthropic-themed UI. Replace localStorage persistence with a `Store` module that calls the REST API (optimistic updates). localStorage kept ONLY for UI prefs (density) + offline read cache.
- **Zero external network.** No cloud. Compliance: McKinsey-sensitive data never leaves the machine.

### Data model (SQLite)
- `epics(id PK, name UNIQUE, is_core, sort_order, created_at)`
- `stories(id PK AUTOINCREMENT, title, epic_id FK NULL=Unassigned, state TEXT[todo|in-progress|blocked|done], urgent INT, points INT, note TEXT, position INT, added_on, completed_on, created_at, updated_at)`
- `comments(id PK, story_id FK, body, created_at)`
- `schema_migrations(version PK, applied_at)`

### REST API (localhost)
- `GET  /api/state` → `{ epics[], stories[] (each with comments[]) }`
- `POST /api/stories` · `PATCH /api/stories/:id` · `DELETE /api/stories/:id`
- `POST /api/stories/reorder` `{ state, orderedIds }`
- `POST /api/comments` `{ storyId, text }`
- `GET/POST /api/epics`
- `POST /api/import` (bulk, transactional) · `GET /api/export`

### Build sequence
**Phase A — Frontend (no backend dependency, usable immediately on current file):**
- A1. Search: toolbar input in static header (NOT `#filter-bar`, which is re-rendered), `searchQuery` state, extra filter pass over title/epic/note/comments, `/` shortcut, clear button, debounce, focus-preservation.
- A2. Compact/density segmented control; persist `rijul-todos-density` (default Comfortable); `.sprint-board.compact` overrides; wrap note+comments in `<details class="panel-collapse">` so compact collapses them, comfortable forces open.
- A3. UI audit fixes: custom `.cs-select` dropdown (replaces modal select + card `.epic-select`, keyboard + ARIA); themed checkbox (`accent-color`/custom); hide number spinner; warm scrollbars; `:focus-visible` rings on all buttons; placeholder → `--muted`; drag-over → `--accent-wash` token; add `--font-sans` token.

**Phase B — Backend (integrates Phase A frontend):**
- B1. Scaffold: `package.json` (`"type":"module"`, start script), `server.js` (http + node:sqlite, 127.0.0.1 bind guard), `db.js` (schema + migrations + repo), `seed.js` (import current 71 stories once).
- B2. Migration/seed: import existing SEED/JSON into DB idempotently; DB owns ids going forward; derive `position` from current order per column.
- B3. Frontend `Store` module: swap localStorage calls for `fetch()`; remove merge/seed/tombstone/runMigrations client logic; keep export/import via API; optimistic UI.
- B4. Auto-backup snapshot on startup + on import.

## Stage 4 — Review findings (to address during build)
1. **Search box destroyed on render** if placed in `#filter-bar` (cleared each render). → Put it in static header; re-apply value + focus after render, or render search bar outside the re-rendered node.
2. **`<details>` collapse changes card markup** → must update `renderCard` and keep comfortable-mode behavior identical (CSS forces open). Verify add-note/add-comment still work in both modes.
3. **ID ownership switch** is a breaking change → Phase B must map old client ids → DB ids for in-flight data; export/import must round-trip. Seed once, then never trust client ids.
4. **Reorder semantics** change from global array to per-column `position` → API `reorder` must persist position; drag code sends ordered id list.
5. **Run friction** (trade-off to confirm): backend means `npm start` (well, `node server.js`) must be running to use the app. Mitigate with a one-command launcher that starts server + opens browser. Single-device only (cross-device is out per compliance).
6. **`node:sqlite` is "experimental"** in Node — pin behavior with tests; keep JSON export as the durable backup so DB is never the only copy.
7. Keep Phase A shippable on its own so a Phase B problem never blocks the usable features.
